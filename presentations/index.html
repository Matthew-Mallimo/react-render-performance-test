Step 1

How traditional SSR works

In classic SSR, the browser asks for a page and the server does all blocking work before it sends the first byte back.

  1. Browser requests the route
  2. Server fetches all required APIs
  3. Server renders the full tree to HTML
  4. HTML is sent as one response

Step 4

Why client fetches are a bad trade

Started later

The browser can’t start the fetch until HTML arrives, JS loads, and the component mounts.

Farther from the data

The client is usually farther away from the data center than your server runtime is.

Worse network conditions

Client networks are less predictable, so the same API often feels slower in the browser.

Result

You improved TTFB, but content completion got worse.

Core problem

You had to choose between early HTML and server-proximate data fetching.

Step 5

The two broken extremes

Traditional SSR

  • Data starts early on the server
  • Browser waits for full completion
  • TTFB is dragged right by the slowest API

Client fetch fallback

  • TTFB improves
  • Slow data starts later in the browser
  • Important content now lands later overall
Old SSR optimizes: completeness before response
Client fetch fallback optimizes: TTFB by deferring work
Step 6

Streaming changes the rule

Streaming SSR keeps data fetching on the server, but removes the requirement that the whole page wait for the slowest part.

  1. Start rendering on the server immediately
  2. Send the shell as soon as it is ready
  3. Keep slow sections behind boundaries
  4. Stream resolved HTML later over the same response

React mechanics

How React does it

  1. Wrap slow UI regions in <Suspense>
  2. Render fallback UI into the shell
  3. Use renderToPipeableStream on the server
  4. Flush when onShellReady fires
  5. Let resolved boundaries stream and hydrate progressively
const { pipe } = renderToPipeableStream(
  <App />,
  {
    onShellReady() {
      res.setHeader('content-type', 'text/html');
      pipe(res);
    }
  }
);
Takeaway

Streaming fixes the false choice

You no longer have to choose between blocking SSR and late client fetching. Start data work on the server, send useful HTML early, and let slow parts arrive later without holding the page hostage.

Keep fetching close to the data Stop making the whole page wait
